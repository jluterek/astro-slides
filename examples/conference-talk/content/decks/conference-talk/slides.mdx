## Where we've been

## The framework treadmill

- Every year a new build tool promised to be **faster**
- Every year the bundle got **bigger** anyway
- We shipped megabytes of JavaScript to render a paragraph
- Meanwhile the platform quietly grew up around us

The browser you're presenting on right now can do things we used to reach for a framework to do.

---

## What the platform grew

- **View Transitions** — animated route changes with no library
- **Container queries** — components that respond to their box, not the viewport
- `oklch()` color — perceptually uniform palettes, native
- **Import maps** and ES modules — real modules, no bundler required
- Streaming HTML — first paint before the data lands

::read::

## The cost of doing nothing

## Ship less, arrive sooner

<Click>Median mobile page ships **~500 KB** of JavaScript</Click>

<Click>Every 100 KB parsed is real main-thread time on a mid-range phone</Click>

<Click>The fastest script is the one you **never send**</Click>

<Clicks>

<div>So the web-native move is subtraction, not addition:</div>

<div>server-render the HTML, hydrate only the islands that need it</div>

<div>and let the platform handle the animation, the routing, the color</div>

</Clicks>

**90%**

less JavaScript when you hydrate islands instead of the whole page

::read::

The 90% figure is the islands-architecture claim: hydrating only the interactive
islands of a content-heavy page ships a small fraction of the JavaScript a
fully-hydrated SPA build sends for the same UI. The honest version is "an order of
magnitude less on content-heavy pages" — measure your own. This block only renders on
the `/read` page: sparse slide live, receipts here.

### The old default

- Ship the whole app as JS
- Framework boots, then renders
- Time-to-interactive waits on the bundle
- Every feature is a dependency

::right::

### The web-native default

- Ship HTML first, JS only where needed
- Renders before it hydrates
- Interactive islands boot independently
- Every feature is a platform primitive

---

## What an island looks like

Only the interactive parts pay the JavaScript cost. Everything else is static HTML.

```ts {2,4-5}
// The counter is an island; the surrounding page is not.
const prerender = true;                // static HTML, zero client JS

import Counter from "./Counter.tsx";   // hydrated on demand
<Counter client:visible />;            // boots only when scrolled into view
```

The `client:visible` directive is the whole trick — defer hydration until the element actually matters.
